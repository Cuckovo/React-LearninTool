# LearnTools — 知识库系统方案

> **版本**: 1.4 | **日期**: 2026-06-21 | **状态**: 方案锁定，可进入开发

---

## 一、项目背景与约束

### 1.1 现有基础设施

| 组件 | 现状 | 说明 |
|------|------|------|
| 数据库引擎 | expo-sqlite (Expo SDK 56 内置) | 已接入，有 `sessions` + `messages` 两张表 |
| ORM | Drizzle ORM + drizzle-zod | 已配置，类型安全 |
| AI 引擎 | DeepSeek Chat API | 当前系统提示词固定为"解题助手"格式 |
| 存储 | SQLite 本地文件 `learntools.db` | 完全离线可用 |
| 路由 | expo-router (3 Tab) | GeoGebra / AI对话(含知识库对话+考核) / 知识库(大纲预览) |

### 1.2 设计约束

1. **离线优先**：所有知识节点数据存储在本地 SQLite，不依赖服务端
2. **最小侵入**：新增表 + 新增模块，`sessions` 表仅新增 `type` 字段（向下兼容），不动 messages 等其余表结构
3. **对话驱动**：用户通过自然语言与 Agent 交互完成所有 CRUD 操作
4. **标准与笔记分离**：每个知识节点的"权威定义"和"用户理解"必须分开存储
5. **标准定义预置**：种子数据中直接写入同济教材权威定义，Agent 教学/考核以此为准，不动态生成

## 二、核心数据模型

### 2.1 新增表：knowledge_nodes

树形结构，通过 `parent_id` 自引用实现无限层级。

```sql
CREATE TABLE knowledge_nodes (
  id                  TEXT PRIMARY KEY,          -- UUID，如 "kn-abc123"
  parent_id           TEXT,                      -- 父节点ID，NULL=根节点
  type                TEXT NOT NULL,             -- 'subject' | 'chapter' | 'concept' | 'note'
  label               TEXT NOT NULL,             -- 显示名称，如 "1.1.1 映射"
  standard_definition TEXT,                      -- 标准定义（教材原文/权威来源）
  user_notes          TEXT,                      -- 用户自定义笔记（对话沉淀）
  mastery_status      TEXT DEFAULT 'not_started',-- not_started | learning | passed | mastered
  metadata            TEXT,                      -- JSON: { tags, difficulty, source, exam_weight }
  sort_order          INTEGER DEFAULT 0,         -- 同级排序
  created_at          INTEGER NOT NULL,
  updated_at          INTEGER NOT NULL
);

CREATE INDEX idx_nodes_parent ON knowledge_nodes(parent_id);
CREATE INDEX idx_nodes_type ON knowledge_nodes(type);
CREATE INDEX idx_nodes_label ON knowledge_nodes(label);
```

### 2.2 字段设计说明

| 字段 | 设计意图 |
|------|---------|
| `type` | 四级分类，控制树的层级语义：subject(学科) > chapter(章) > concept(节/知识点) > note(自定义笔记) |
| `standard_definition` | **只由 Agent 写入**，用户不可直接编辑。来源：教材原文 / 权威数学定义。Agent 出题、考核时以此为准 |
| `user_notes` | **对话过程中由 Agent 总结生成**，用户可通过对话修改。记录用户用自己的话对概念的重新表述、记忆技巧、易错点 |
| `mastery_status` | 学习进度状态机：not_started → learning(对话中) → passed(考核通过) → mastered(多次复习确认) |
| `metadata` | JSON 扩展字段，存储标签、难度、考频等非结构化元数据 |

### 2.4 sessions 表变更（Q2 决策：独立类型）

> **决策 Q2 — sessions 独立类型**：`sessions` 表新增 `type` 字段区分解题模式与知识库模式。

```sql
-- Migration SQL
ALTER TABLE sessions ADD COLUMN type TEXT DEFAULT 'solver';
-- type = 'solver'    → 解题模式会话（默认兼容存量数据）
-- type = 'knowledge' → 知识库学习会话
```

**Drizzle Schema 变更**：

```typescript
// src/db/schema.ts — sessions 表更新
export const sessions = sqliteTable('sessions', {
  id: text('id').primaryKey(),
  title: text('title').notNull(),
  type: text('type', { enum: ['solver', 'knowledge'] }).default('solver').notNull(),  // ← 新增
  createdAt: integer('created_at').notNull(),
  updatedAt: integer('updated_at').notNull(),
});
```

**影响范围**：
- `src/db/repository.ts` — `createSession()` 增加 `type` 参数
- `src/lib/app-state.tsx` — 会话列表可按 `type` 分组/过滤
- 历史面板 UI — 解题会话和知识库会话用不同图标区分（📐 vs 📖）
- **向下兼容**：`DEFAULT 'solver'` 保证存量会话不受影响

#### 会话生命周期关联

```
用户点击 1.1.1 映射「去学习」
        │
        ▼
createSession({ title: "学习: 1.1.1 映射", type: "knowledge" })
        │
        ▼
AI 对话页绑定该 session → 知识库模式
        │
        ▼
考核通过后
  → knowledge_nodes 的 metadata.source_session_id = session.id
  → 可追溯"这个笔记来自哪次学习"
```

### 2.5 与现有表的关系

```
sessions (type: solver/knowledge) ──1:N──> messages     (迁移变更：新增 type 字段)
                                    │
knowledge_nodes <────────────────────┘       (新增，通过 metadata.source_session_id 追溯)
                                    │
agent_logs    <─────────────────────┘       (新增，记录所有 Agent 的 DB 操作)
```

> **关联方式**：`knowledge_nodes` 不与 `sessions` 直接外键关联。通过 `metadata` JSON 字段中的 `source_session_id` 追溯"这个笔记是从哪次学习会话中沉淀的"。知识节点的生命周期远超单次会话。

### 3.1.2 上下文隔离策略

> **核心决策：解题模式与知识库模式完全隔离；知识库内部教学与考核共享同一 session。**

| 切换场景 | 策略 | 原理 |
|---------|------|------|
| 解题 → 知识库 | **新建 session + 切换提示词** | 解题上下文（【解题过程】【图像判断】等格式）不能污染知识库教学 |
| 知识库 → 解题 | **新建 session + 切换提示词** | 同上 |
| 知识库内：教学 → 考核 | **沿用同一 session** | 考核需要前置教学上下文才能精准评分 |
| 知识库内：考核 → 沉淀 | **沿用同一 session** | 沉淀依赖考核结果和教学对话 |
| 跳转新知识点 | **新建 session**（Q3 决策） | 每个知识点独立学习上下文 |

#### 上下文注入机制

知识库模式下的首条 system message 自动注入：

```json
{
  "role": "system",
  "content": "## 当前学习节点\n- 知识点：1.1.1 映射\n- 标准定义：设X、Y是两个非空集合。如果存在一个法则f，使得对X中的每个元素x，按法则f，在Y中有唯一确定的元素y与之对应，则称f为从X到Y的映射，记作 f: X→Y。\n- 掌握状态：not_started\n\n## 可用数据库操作\n你可以通过以下格式操作知识库数据库：\n- 写入用户笔记: db.setUserNotes(nodeId, text)\n- 更新掌握状态: db.updateMastery(nodeId, status)\n- 查询子节点: db.getChildrenOf(nodeId)"
}
```

> 这条 system message 只在 session 创建时注入一条，后续对话消息正常追加。

#### 3.2.4 个人笔记编辑模式

```
阅览模式:
  ┌──────────────────────────────┐
  │ 映射=每个输入只有一个输出...  │  ← 普通 Text
  └──────────────────────────────┘
         │ 单次点击
         ▼
编辑模式:
  ┌──────────────────────────────┐
  │ ┌──────────────────────────┐ │
  │ │ 映射=每个输入只有一个... │ │  ← TextInput multiline
  │ │                          │ │     border 高亮 (brand)
  │ │                          │ │     autoFocus
  │ └──────────────────────────┘ │
  │              已自动保存 ✓    │
  └──────────────────────────────┘
         │ 键盘收起
         ▼
  → 恢复阅览模式，笔记已持久化到 SQLite
```

**编辑细节**：
- 单击笔记区域 → `TextInput` 进入编辑，自动获取焦点
- 输入后 1 秒防抖自动保存（`knowledgeService.setUserNotes(nodeId, text)`）
- 键盘收起（`Keyboard.dismiss` / 点击空白 / 物理返回键）→ 恢复阅览
- 内容未修改则不触发保存

## 四、系统提示词设计

> 这是整套方案最核心的部分。提示词的质量直接决定 Agent 的行为是否符合预期。

### 4.1 设计原则

1. **一份提示词覆盖全闭环**：教学、考核、沉淀三个阶段用同一个 prompt，不拆分。大模型根据对话进展自行判断当前处于哪个阶段
2. **数据库操作通过结构化指令注入**：在 system message 中明确告知 Agent 可用的 DB 操作及其调用格式
3. **考核走第二轮 API**：教学/沉淀由主提示词覆盖；考核评分因为需要独立的评分逻辑（0-100 合理度），走单独的 scoring prompt
4. **Agent 不直接操作数据库**：Agent 输出结构化 JSON 指令 → 前端 KnowledgeService 解析执行 → 返回结果给 Agent 下一条消息

### 4.2 解题模式提示词（保持现有，小改）

```
你是高等数学 AI 解题助手。请严格按以下结构回复：

【解题过程】
分步解答，公式用 $...$（行内）和 $$...$$（块级）包裹。使用中文。

【图像判断】
可绘制 / 不可绘制

【函数表达式】
f(x) = sin(x)    （如果可绘制，使用 GeoGebra 兼容语法）
无                （如果不可绘制）
```

> 不变。与知识库模式完全隔离。

### 4.3 知识库助手提示词（统合：教学 + 考核 + 沉淀 + DB 操作）

```
# 角色定义
你是「伴学知识库」的高等数学学习助手。基于同济大学《高等数学》（第七版），
帮助用户学习概念、检测理解、沉淀笔记。

# 当前学习上下文
你会在对话开始时收到一条系统消息，包含：
- 当前知识节点（label + standard_definition）
- 用户的掌握状态
- 可用的数据库操作指令

# 行为准则

## 教学阶段
1. 当用户提问时，从标准定义出发展开讲解
2. 用类比和举例辅助理解，举例不要偏离标准定义
3. 如果用户表现出困惑，换一种方式讲解而非简单重复
4. 概念辨析（如映射 vs 函数、无穷小 vs 无穷大）是你的强项
5. 当用户说"明白了""懂了""原来如此"等理解信号，且你已经讲解超过 2 轮，
   主动提议："要不要我出两道题检验一下你的理解？"

## 考核阶段
6. 用户说"考考我""测一下""来几道题"时触发考核；
   或你主动提议后用户同意，也触发考核
7. 出 2-3 道题，覆盖：概念辨析、举例/反例、简单应用
8. 用户回答后，**不要自己判断对错**。而是调用评分指令
9. 评分完成后，将结果反馈给用户：
   - 告知合理度百分比 + 通过/未通过
   - 指出薄弱点
   - 通过 → 进入沉淀阶段
   - 未通过 → 回到教学阶段，针对薄弱点补充

## 沉淀阶段
10. 通过后，基于对话内容生成用户笔记摘要
11. 用用户自己的语言风格，不要照搬教材
12. 展示给用户确认："我帮你这样记，可以吗？"
13. 用户确认后，调用 DB 指令写入 user_notes + 更新 mastery_status

# 数据库操作指令
当需要操作知识库时，在回复末尾以以下格式输出指令（不展示给用户）：

```db
{
  "action": "set_user_notes",
  "nodeId": "kn-ch-1-1-1",
  "value": "映射=每个输入只有一个输出，但值域不一定全被占满。函数是特殊映射（数→数）。满射=值域全被占满的映射。"
}
```

可用指令：
- 写入笔记: `{ "action": "set_user_notes", "nodeId": "...", "value": "..." }`
- 更新状态: `{ "action": "set_mastery", "nodeId": "...", "value": "passed"|"learning" }`
- 查询节点: `{ "action": "get_node", "nodeId": "..." }`
- 查询子节点: `{ "action": "get_children", "parentId": "..." }`

注意：
- 写入 user_notes 前必须先向用户确认内容，确认后立即执行 DB 指令
- 更新 mastery_status 为 passed 只在考核通过 + 用户确认笔记后执行
- DB 指令块使用 ```db 代码块包裹，由前端解析执行

# 禁止行为
- 不要编造标准定义（当前节点的 standard_definition 已提供）
- 不要替代用户做判断（笔记必须确认）

// src/lib/assessment-prompt.ts

export const ASSESSMENT_SYSTEM_PROMPT = `你是高等数学考核评分专家。
你的任务是比较学生的回答与标准定义的语义合理度，给出 0-100 的分数。

评分原则：
- 基于理解深度，不要求逐字对照
- 核心概念（如"每个输入对应唯一输出"）正确 → 基础分 60+
- 能举出恰当的例子 → +10-15 分
- 能辨析易混淆概念 → +10-15 分
- 有概念性错误 → 根据严重程度扣分
- 60 分及以上为"通过"

输出格式（严格 JSON，不要额外文字）：
{

  "analysis": "用户正确理解了映射的核心概念...",
  "weak_points": ["满射概念混淆"],
  "passed": true,
  "suggested_user_note": "建议的用户笔记总结"
}`;

export function buildAssessmentPrompt(params: {
  standardDefinition: string;
  questions: string[];
  userAnswers: string[];
}): string {
  return `## 标准定义
${params.standardDefinition}

## 考核问题与学生回答
${params.questions.map((q, i) => `### 问题 ${i + 1}
${q}

### 学生回答
${params.userAnswers[i] || '(未作答)'}
`).join('\n')}

请评分并输出 JSON。`;
}
```

### 4.6 DB 指令解析 (Q8 决策)

> **决策 Q8：```db 指令块放在 AI 回复末尾，前端正则过滤后不展示。**

前端在收到 AI 回复后，扫描 ```db 代码块并执行对应操作。展示给用户前，正则移除 db 代码块：

```typescript
// src/lib/db-command-parser.ts

interface DBCommand {
  action: 'set_user_notes' | 'set_mastery' | 'get_node' | 'get_children';
  nodeId: string;
  value?: string;
  parentId?: string;
}

export function extractDBCommands(content: string): DBCommand[] {
  const regex = /```db\n([\s\S]*?)\n```/g;
  const commands: DBCommand[] = [];
  let match;
  while ((match = regex.exec(content)) !== null) {
    try {
      commands.push(JSON.parse(match[1]));
    } catch { /* ignore malformed */ }
  }
  return commands;
}

## 七、与现有 proposal 的差异说明

| 维度 | DB-Migration-Proposal.md | 本方案 |
|------|--------------------------|--------|
| knowledge_nodes 字段 | `content` (单一字段) | `standard_definition` + `user_notes` (分离) |
| standard_definition | 无 | **种子数据预置教材权威定义** |
| mastery 状态 | 无 | `mastery_status` 四态状态机 |
| 节点类型 | `subject/chapter/concept/formula/note` (5类) | 简化至 `subject/chapter/section/concept` (4类) |
| sessions 表 | 无类型区分 | **新增 `type` 字段 (`solver` | `knowledge`)** |
| 会话关联 | 无 | `metadata.source_session_id` 可选追溯 |
| agent_logs | 有 | 保留，用于审计 |
| 考核方式 | 未涉及 | **百分比合理度评分** (大模型语义比较) |
| 页面架构 | 独立知识库对话 Tab | **知识库 Tab = 大纲预览，对话/考核在 AI 对话页** |
| "去学习"跳转 | 未涉及 | **每次新建独立 `type: knowledge` 会话** |

### 关键改动理由

**为什么分离 standard_definition 和 user_notes？**
- Agent 考核时需要基于权威定义出题，复习时需要看用户自己的理解
- 两者不能混在一个字段里（否则考核时无法区分哪些是教材内容、哪些是用户自己写的可能有偏差的理解）

**为什么用百分比而不是二元判断？**
- 数学概念理解是连续的，不是简单的"懂/不懂"二分
- 80% 合理度和 50% 合理度对后续教学策略完全不同
- 用户可以直观看到自己的理解程度，有成就感
